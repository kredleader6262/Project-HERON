### Divergences from v3/Stack Context

- Charting is Chart.js, not Plotly or lightweight-charts.
- Trading-calendar dependency from v3 is absent; the code uses custom NY trading-day helpers and `pytz`.
- Broker implementation is paper-only even though docs discuss a margin account/live enforcement. Live mode exists as a runtime/risk mode, but broker adapter isolation is not complete for a live Alpaca account.
- Schema migrations are custom idempotent Python/SQL, not Alembic.
- The dashboard is much broader than v3 Dashboard v1, but all routes remain in one module.
- Claude model IDs are env-configurable and currently default to `claude-sonnet-4-6` and `claude-haiku-4-5-20251001`, not fixed dated IDs from older docs.
- The EOD debrief calls Claude and returns token/cost data, but it does not persist that cost via `log_cost` in the current code path ([heron/alerts/debrief.py](../heron/alerts/debrief.py#L94-L128), [heron/alerts/debrief.py](../heron/alerts/debrief.py#L153-L178)).

## Section F - Campaigns: Current Reality

Campaigns are real persisted rows, not logical wrappers. The main evidence is the `campaigns` and `campaign_state_log` DDL in the journal schema ([heron/journal/__init__.py](../heron/journal/__init__.py#L16-L44)) and the service layer in `heron/journal/campaigns.py` ([heron/journal/campaigns.py](../heron/journal/campaigns.py#L1-L145)).

Current campaign fields:

- Identity/text: `id`, `name`, `description`.
- Mode/state: `mode` (`paper` or `live`), `state`.
- Allocation/window: `capital_allocation_usd`, `paper_window_days`.
- Lineage/timestamps: `parent_campaign_id`, `started_at`, `graduated_at`, `retired_at`, `retired_reason`, `created_at`, `updated_at`.

Current campaign states and transitions:

| From | To |
|---|---|
| `DRAFT` | `ACTIVE`, `RETIRED` |
| `ACTIVE` | `PAUSED`, `GRADUATED`, `RETIRED` |
| `PAUSED` | `ACTIVE`, `RETIRED` |
| `GRADUATED` | `RETIRED` |
| `RETIRED` | `DRAFT` |

Service-layer functions:

- `create_campaign(conn, id, name, ..., state='DRAFT')`

## Section G - Candidates: Current Reality

### Origin

Candidates are primarily produced by the Research layer:

1. `ResearchPass.run()` fetches watchlist news via `DataFeed`.
2. It classifies articles with the local Ollama/Qwen classifier.
3. It gathers best-effort quote context.
4. It calls `generate_candidates()`.
5. If candidates exist and `ANTHROPIC_API_KEY` is configured, it escalates some candidates to Claude ([heron/research/orchestrator.py](../heron/research/orchestrator.py#L47-L156)).

`generate_candidates()` filters relevant classifications, watchlist tickers, duplicate recent pending tickers for the same strategy, and score thresholds, then writes `source='research_local'` candidates through the journal helper ([heron/research/candidates.py](../heron/research/candidates.py#L28-L101)).

Manual/demo paths also create candidates through `heron.journal.candidates.create_candidate()`. The schema allows `source` values such as `research_local`, `research_api`, and `manual`, but this is not enforced by a CHECK constraint.

### Fields

Current candidate columns are:

`id`, `strategy_id`, `ticker`, `side`, `source`, `local_score`, `api_score`, `final_score`, `disposition`, `rejection_reason`, `thesis`, `context_json`, `created_at`, `disposed_at`.

The helper `create_candidate()` writes pending candidates with optional scores, thesis, and JSON context ([heron/journal/candidates.py](../heron/journal/candidates.py#L1-L19)).

Candidate context currently stores article/classifier fields:

- `article_id`
- `relevance_score`
- `sentiment`
- `sentiment_score`
- `category`
- `source='local_classifier'`
- optional `price_context`

It does not store a `signal_id` or shared upstream entity.

### Downstream Consumers

| Consumer | How it uses candidates |
|---|---|
| Dashboard `/candidates` and `/candidate/<id>` | Shows queue, thesis/context, related audits; filters by strategy state/mode ([heron/dashboard/__init__.py](../heron/dashboard/__init__.py#L366-L388), [heron/dashboard/__init__.py](../heron/dashboard/__init__.py#L698-L713)). |
| Dashboard accept/reject routes | Change disposition to `accepted` or `rejected` ([heron/dashboard/__init__.py](../heron/dashboard/__init__.py#L1412-L1435)). |
| Executor cycle | Looks for `accepted` candidates per strategy, skips if already traded, requires market data context, computes levels, calls `Executor.enter_position()`, and rejects on risk/level failure ([heron/execution/cycle.py](../heron/execution/cycle.py#L135-L226)). |
| Trades journal | `trades.candidate_id` links executed trades back to the candidate. |
| Baseline runner | `mirror_candidate_to_baseline()` creates a separate baseline candidate row with `source='baseline_mirror'` and `context_json.mirrored_from` ([heron/strategy/baseline.py](../heron/strategy/baseline.py#L54-L96)). |
| Claude thesis/escalation/audit | Reads candidate context/scores; writes `api_score` and enriched thesis; logs sampling or post-mortem audits. |

### Shared Candidate Concept

There is no shared candidate concept today. A candidate is strictly attached to one `strategy_id`. Baseline comparison duplicates candidates into a second candidate row for the baseline strategy rather than sharing one candidate across variants. This is important for a future Signal layer: current code assumes Candidate -> Strategy, not Signal -> many Strategies.

### Dispositions

The schema default is `pending`. `dispose_candidate()` permits only `accepted`, `rejected`, or `expired` ([heron/journal/candidates.py](../heron/journal/candidates.py#L21-L28)). Dashboard exposes accept/reject. Expiry exists in the helper and tests, but no dashboard expiry route was found.

## Section H - Trading Safety Surface

This is the area most likely to cause real-money regressions if refactored casually.

### Pre-Trade Checks

Pre-trade checks live in `heron/strategy/risk.py`. The composite `pre_trade_checks()` returns all check results and logs failures to `events` ([heron/strategy/risk.py](../heron/strategy/risk.py#L212-L256)). Checks include:

- `system_mode`: blocks entries in global `SAFE` mode.
- `wash_sale`: paper mode no-op; live mode checks active lots by ticker family.
- `pdt`: paper mode no-op; live mode blocks same-day-required entries when count is at cap.
- `exposure`: mode-scoped total exposure cap.
- `portfolio_cap`: per-strategy allocation cap from `strategy.portfolio`.
- `positions`: mode-scoped open position count.
- `daily_entries`: NY trading-day entry cap.
- `daily_loss`: NY trading-day realized loss cap.
- `single_trade`: stop-vs-entry risk cap.
- `quote_fresh`: quote age cap.

Wash-sale/PDT storage and queries live in `heron/journal/trades.py`. Losses create `wash_sale_lots` on close, same-day closes create `pdt_daytrades`, and both have mode-aware query paths ([heron/journal/trades.py](../heron/journal/trades.py#L57-L94), [heron/journal/trades.py](../heron/journal/trades.py#L180-L244)).

### Idempotent Order IDs

Order ID helpers live in `heron/execution/broker.py`:

- `make_entry_order_id(strategy_id, candidate_id, ticker, side)` uses candidate ID as deterministic nonce when present.
- `make_close_order_id(strategy_id, trade_id, ticker, side)` uses trade ID as deterministic nonce.
- IDs are opaque; code warns not to parse on underscores ([heron/execution/broker.py](../heron/execution/broker.py#L9-L45)).

`Executor.enter_position()` uses `make_entry_order_id()` unless an explicit ID is passed. `_close_position()` uses `make_close_order_id()` for every close attempt ([heron/execution/executor.py](../heron/execution/executor.py#L95-L111), [heron/execution/executor.py](../heron/execution/executor.py#L229-L238)).

### Broker Submission Isolation

- The abstract interface is `BrokerAdapter` ([heron/execution/broker.py](../heron/execution/broker.py#L47-L88)).
- Alpaca paper trading is isolated in `AlpacaPaperAdapter`. It wraps `alpaca-py`, uses DAY TIF, and handles Alpaca duplicate `client_order_id` API errors as existing orders ([heron/execution/alpaca_adapter.py](../heron/execution/alpaca_adapter.py#L16-L171)).
- `Executor._submit_with_retry()` catches submission exceptions, logs a `submit_error`, checks `broker.get_order(client_order_id)`, and treats an existing order as success before re-raising ([heron/execution/executor.py](../heron/execution/executor.py#L124-L149)).

### Executor Polling and Virtual Stops

Entry flow in `Executor.enter_position()` is quote -> DERISK scaling -> `pre_trade_checks()` -> submit -> create trade -> short fill poll -> fill journal ([heron/execution/executor.py](../heron/execution/executor.py#L39-L121)). `_await_fill()` polls `get_order()` for up to 3 seconds at 0.5 second intervals ([heron/execution/executor.py](../heron/execution/executor.py#L151-L169)).

Virtual stops/targets are polled in `Executor.check_exits(strategy)`: for each filled open trade, it gets a fresh quote, computes current price and days held, asks the strategy `should_exit()`, checks min hold, and calls `_close_position()` ([heron/execution/executor.py](../heron/execution/executor.py#L171-L219)). The runtime executor cycle is scheduled every five minutes during market hours ([heron/runtime/supervisor.py](../heron/runtime/supervisor.py#L36-L43)).

### Reconciliation

There are two reconciliation-like surfaces:

- `Executor.reconcile()` compares broker positions with filled open journal trades and logs `reconciliation_drift` events ([heron/execution/executor.py](../heron/execution/executor.py#L256-L320)).
- `resilience.startup_audit.run_startup_audit()` optionally reconciles broker vs journal, checks stop coverage, counts pending work, logs a `startup_audit` event, and returns `clean` or `drift` ([heron/resilience/startup_audit.py](../heron/resilience/startup_audit.py#L1-L75)).

`runtime.preflight.preflight()` blocks live supervisor startup if startup audit is not clean; paper mode reports audit issues as warnings ([heron/runtime/preflight.py](../heron/runtime/preflight.py#L1-L58)).

### Kill Switches and Halts

| Mechanism | Location | Effect |
|---|---|---|
| Stale quote | Executor and risk check | Rejects entry when quote is stale. |
| System `SAFE` mode | `strategy.policy` + `strategy.risk` | Blocks all new entries; exits/reconciliation still run. |
| System `DERISK` mode | `strategy.policy` + executor | Scales quantity before risk checks. |
| Cost cap | `research.cost_guard` | Halts Research when cap/projected cap is tripped; Strategy/Execution continue. |
| Live preflight blockers | `runtime.preflight` | Blocks supervisor start in live mode for missing required secrets or startup audit drift. |
| Promotion gate | Dashboard promote route | Blocks PAPER -> LIVE when monthly review missing or latest parity verdict missing/failing, unless force override for parity only ([heron/dashboard/__init__.py](../heron/dashboard/__init__.py#L1343-L1396)). |
| Dashboard/operator actions | `/actions` and `/scheduler` command aliases | Queue run-now/pause/resume commands, not direct kill switch, but operational control surface. |

## Section I - Baseline-Beat Machinery

A deterministic baseline runner already exists in `heron/strategy/baseline.py`. It creates a baseline strategy row named `<parent>_baseline`, copies config/template/campaign, marks `is_baseline=1`, and links `parent_id` ([heron/strategy/baseline.py](../heron/strategy/baseline.py#L16-L52)).

Candidate mirroring exists, but it creates a separate baseline candidate row; it does not share the same candidate row ([heron/strategy/baseline.py](../heron/strategy/baseline.py#L54-L96)).

The paired bootstrap CI test is implemented twice as a shared primitive:

- `heron/backtest/significance.bootstrap_beat_test()` is the pure paired bootstrap helper ([heron/backtest/significance.py](../heron/backtest/significance.py#L1-L57)).
- `strategy.baseline.run_beat_test()` computes paired daily return diffs from live/paper `trades` rows and calls that helper ([heron/strategy/baseline.py](../heron/strategy/baseline.py#L98-L188)).
- `backtest.parity.compute_parity_report()` compares saved backtest equity curves and persists parity into `metrics_json['parity']` during report save ([heron/backtest/parity.py](../heron/backtest/parity.py#L1-L72), [heron/backtest/report.py](../heron/backtest/report.py#L23-L80)).

Equity curve storage:

- Live/paper strategy equity curves are computed on demand from closed trades by `get_equity_curve()`.
- Backtest equity curves are stored inside `backtest_reports.metrics_json` when `save_report()` runs.
- There is no separate equity-curve table.

Baseline scope:

- Baselines are per-strategy (`parent_id`, `is_baseline`, `<id>_baseline`).
- Baselines copy `campaign_id`, so variants can belong to one campaign.
- There is no per-campaign aggregation or campaign-level baseline-beat verdict today.

## Section N - Risk Areas for the Refactor

### UI Nav Restructure

Risk is compatibility and operator muscle memory, not just template edits. Tests and README currently depend on specific legacy routes such as `/overview`, `/health`, `/scheduler`, `/actions`, and `/campaigns`; route aliases should be treated as behavior until intentionally retired.

### Campaign -> Desk Renaming

The table already does most Desk-like work: persisted identity, state, mode, capital, window, lineage, and strategy grouping. The risk is adding a parallel `desks` model and splitting ownership; evidence favors renaming/relabeling only unless v4 Desk semantics require fields not present today.

### Candidate -> Signal Layering

Current code assumes candidates belong to exactly one strategy. Baseline machinery duplicates candidates instead of sharing them. A Signal layer must be additive/upstream or it will break candidate approval, executor, trade FK, audit, and baseline mirroring assumptions.

### Approvals Page Redesign

Approvals are currently scattered: strategy proposals, candidate accept/reject, monthly reviews, parity/promotion gates, policy alerts, contamination findings, and action commands live on different pages. Consolidation risks bypassing existing gate logic if UI handlers are rewritten instead of reusing service-layer functions.

### Activity Timeline Consolidation

`events`, `scheduler_runs`, audits, trades, candidates, reviews, and campaign/strategy state logs are separate streams. `/actions` already merges `events` and `scheduler_runs`; expanding this requires timestamp normalization and careful source labeling so operator decisions remain auditable.

### Portfolio Attribution Rebuild

Portfolio allocation exists, but campaign/desk-level attribution does not. Trades point to strategies, strategies point to campaigns, and backtest curves store per-strategy data; any desk attribution will need joins/computation unless new summary tables are introduced.

### System Section Split

System-like pages currently include Actions, Resilience, Costs, Policies, Agents status, Setup, Glossary, Health, and maybe Audits. The risk is burying money-safety surfaces like cost trip, startup audit, SAFE mode, and scheduler controls under cosmetic categories.

### Anything Money-Moving

All entry safety depends on `Executor.enter_position()` invoking stale quote checks, DERISK scaling, `pre_trade_checks()`, deterministic order IDs, and broker submit retry semantics. Refactors must not create alternate order paths through the adapter or CLI/dashboard helpers.

### Untested Refactor-Touched Areas

Dashboard tests are smoke-level, HTMX behavior is barely tested, executor-cycle full candidate-to-order flow is thin, real Alpaca integration is untested, and campaign-level aggregation is absent. Any v4 work in these areas should add focused compatibility and service tests before/with route or schema changes.

1. Desk reuses the existing `campaigns` table; no separate persisted Desk entity in Phase 1.
2. Internal database/table/column names remain `campaigns` and `campaign_id`; operator-facing UI/docs say Desk.
3. Desk-level performance metrics and equity curves are computed on demand first; summary/cache tables are Phase 2 only if real dashboard latency requires them.
4. One Signal can produce candidates for many strategies.
5. Bearish or negative-bias Signals are preserved as informational Signals even while current strategies are long-only.
6. Candidate approval remains per-strategy in Phase 1; shared signal approval is deferred to Phase 2.
7. Old dashboard routes remain as compatibility aliases throughout the refactor unless a later stage explicitly retires one.
8. Audits appear in both places by intent: chronological audit events in Activity, audit/model logs in System -> Introspection.
9. Agents are operator-visible under System -> Introspection and filterable in Activity, not a top-level nav section.
10. Activity is a new aggregate page. `/actions` remains the scheduler-control alias and is not replaced.
11. The 90-day paper window is market days for gating. The current calendar-day `days_active()` helper is a known divergence to fix in Stage 7.
12. Live Alpaca adapter hardening stays outside the v4 Desk/Signal refactor as a separate project.
13. EOD debrief Claude calls should be charged to `cost_tracking`; current behavior is a Stage 7 divergence.

15. Desk/Signal terminology applies to dashboard/docs in Phase 1. CLI keeps `campaign` and `candidate` naming, with possible aliases later.
